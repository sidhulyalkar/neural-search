# Corpus QA Workflow

Neural Search tracks dataset-card and ingestion quality with an explicit QA status plus reviewer fields. The current implementation persists review state in `data/qa/dataset_qa.json` so it works with the demo seed corpus and local API without requiring a database migration step.

## QA Statuses

- `unreviewed`: imported but not inspected.
- `auto_generated`: generated by deterministic extraction/card tooling.
- `needs_review`: human review found a gap or uncertainty.
- `reviewed`: labels and card content have been checked.
- `trusted`: reviewed and ready for demos or higher-confidence retrieval.
- `rejected`: should not be used in demos or trusted exports.

## Review Fields

Each dataset card can record:

- `task_labels_verified`
- `modality_labels_verified`
- `behavior_labels_verified`
- `brain_regions_verified`
- `linked_papers_verified`
- `notebook_tested`
- `reviewer_notes`

## CLI

List datasets still awaiting acceptance:

```bash
python -m neural_search.qa.list_unreviewed
```

Mark a dataset reviewed, trusted, or rejected:

```bash
python -m neural_search.qa.mark_reviewed --dataset-id DEMO_GONOGO_CALCIUM
python -m neural_search.qa.mark_trusted --dataset-id DEMO_GONOGO_CALCIUM
python -m neural_search.qa.mark_rejected --dataset-id DEMO_GONOGO_CALCIUM --notes "Insufficient task metadata"
```

Export accepted cards:

```bash
python -m neural_search.qa.export_reviewed_cards
```

Generate the corpus report:

```bash
python -m neural_search.reports.dataset_compilation
```

The report includes reviewed/trusted/rejected counts, common missing metadata, and the top datasets ready for demos.

## Frontend Review

Dataset result cards display a QA badge. Search results can be filtered to all datasets, reviewed datasets, or trusted datasets. Dataset detail pages include a Corpus QA panel for updating status, verification checkboxes, notebook testing, and reviewer notes.

## Persistence Notes

The JSON QA state is intentionally small and source-control friendly. When Neural Search switches the demo API to database persistence, the matching SQLAlchemy fields and Alembic migration are already present on `datasets` and `dataset_cards`.
